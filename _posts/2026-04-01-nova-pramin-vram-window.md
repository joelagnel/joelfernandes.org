There is a chicken-and-egg problem buried inside every NVIDIA GPU driver: to initialize the GPU's
MMU and page tables, you need to write data into VRAM. But VRAM normally becomes accessible
*through* the MMU. So how do you write anything before the MMU exists?

The answer is PRAMIN -- a hardware aperture that has been in NVIDIA GPUs for decades. I spent the
last several months adding PRAMIN support to nova-core, the new Rust-based Linux GPU driver for
NVIDIA hardware. This post covers what PRAMIN is, how the aperture actually works at the register
level, and why implementing it in Rust led to some interesting design choices.

The v8 patch series landed on the mailing lists on February 24, 2026.[[1]](#ref1)

## What Is PRAMIN

PRAMIN is a 1MB window into VRAM, accessible through BAR0 (the GPU's primary MMIO BAR). The
window sits at a fixed offset in BAR0 -- `0x700000` -- and its position within the VRAM address
space is controlled by a single register: `NV_PBUS_BAR0_WINDOW` at offset `0x1700`.

The register layout:

```
NV_PBUS_BAR0_WINDOW (offset 0x1700):
+-------+--------+--------------------------------------------+
| 31:26 | 25:24  | 23:0                                       |
| RSVD  | TARGET | BASE_ADDR (bits 39:16 of VRAM address)     |
+-------+--------+--------------------------------------------+

TARGET values:
  0x0 = VID_MEM   (VRAM -- the only value nova uses)
  0x1 = SYS_MEM_COH
  0x2 = SYS_MEM_NONCOH
```

The `BASE_ADDR` field holds bits `[39:16]` of the VRAM address. Because it occupies bits `[23:0]`
of the register and encodes 40-bit addresses shifted right by 16, window placement must be 64KB
aligned. You can slide the window anywhere in a 1TB address space, 64KB at a time.

A read or write to `BAR0 + 0x700000 + offset` reaches `VRAM[(BASE_ADDR << 16) + offset]`.

```
   CPU
    |
    | MMIO
    v
+--------+        +---------------------+
|  BAR0  |------->| PBUS (GPU)          |
+--------+        |                     |
                  |  [0x700000 .. +1MB] |-----> VRAM[BASE_ADDR<<16 .. +1MB]
                  |                     |
                  | NV_PBUS_BAR0_WINDOW |
                  | (controls BASE_ADDR)|
                  +---------------------+
```

The driver programs `BASE_ADDR`, then does plain MMIO reads and writes into the window region of
BAR0. The GPU hardware routes them to the right VRAM address. No MMU, no page tables required.

## Why Nova Needs This

nova-core is the first-level NVIDIA GPU driver in the kernel -- the part that talks directly to
hardware to initialize the GPU and bring up the GSP firmware.[[2]](#ref2) Getting the GPU
memory management working requires writing GPU page table entries into VRAM. The page table
structures live in VRAM. The MMU reads from VRAM to walk them. So PRAMIN is the only sane path
for bootstrap.

The driver also uses PRAMIN to write data structures that GSP firmware needs during
initialization -- before any GPU-side virtual memory mapping is alive.

## The Rust Implementation

The PRAMIN code in nova-core is in `drivers/gpu/nova-core/mm/pramin.rs`.[[1]](#ref1) The design
uses two types:

- `Pramin` -- owns the BAR0 reference and an internal mutex protecting window state
- `PraminWindow` -- a guard that holds the mutex and provides the actual read/write methods

```rust
#[pin_data]
pub(crate) struct Pramin {
    bar: Arc<Devres<Bar0>>,
    #[pin]
    state: Mutex<PraminState>,
}

pub(crate) struct PraminWindow<'a> {
    bar: Arc<Devres<Bar0>>,
    state: MutexGuard<'a, PraminState>,
    saved_base: usize,
}
```

`PraminWindow` holds a `MutexGuard`, so only one window can exist at a time -- enforced by the
type system at compile time, not by a runtime check. Drop restores the saved window base, which
matters because other parts of the driver might rely on the window position being stable.

The accessors are generated by macros:

```rust
macro_rules! define_pramin_read {
    ($name:ident, $ty:ty) => {
        pub(crate) fn $name(&mut self, vram_offset: usize) -> Result<$ty> {
            let (bar_offset, new_base) =
                self.compute_window(vram_offset, core::mem::size_of::<$ty>())?;
            let bar = self.bar.try_access().ok_or(ENODEV)?;
            if let Some(base) = new_base {
                Self::write_window_base(&bar, base);
                self.state.current_base = base;
            }
            bar.$name(bar_offset)
        }
    };
}
```

When you call `window.try_read32(0x200000)`, the code figures out whether the address falls within
the current 1MB window. If not, it repositions the window by writing a new `BASE_ADDR` to
`NV_PBUS_BAR0_WINDOW` and then performs the MMIO access. The repositioning is transparent to the
caller.

One non-obvious constraint in the mutex comment: the PRAMIN lock must never be held while
performing `GFP_KERNEL` allocations. The memory reclaim path can call `dma_fence_wait()`, and
if that fence signals through a path that needs the PRAMIN lock, you have a deadlock. This is
called out explicitly in the code comment since it is easy to violate.

## Self-Tests

The series also adds self-tests (`drivers/gpu/nova-core/mm/pramin.rs` test module) that exercise
PRAMIN write/read-back across window boundaries -- including cases where consecutive writes land
in different 1MB windows, forcing the auto-reposition logic to run. These run in a virtme-ng VM
with GPU passthrough before any series is posted.

## What Is Next

The v8 series adds PRAMIN plus a full GPU memory management stack: a buddy allocator over usable
VRAM, MMU v2/v3 page table types, a TLB flush interface, a Virtual Memory Manager, and a BAR1
aperture layer for user-space GPU memory mapping.[[1]](#ref1) PRAMIN is the lowest layer -- the
one that everything else depends on to get data into VRAM during early init.

Extending PRAMIN to Hopper and Blackwell requires a different register (`NV_XAL_EP_BAR0_WINDOW`
at `0x00117750`) instead of `NV_PBUS_BAR0_WINDOW`. The per-generation dispatch is in the works.

## Key Takeaways

- PRAMIN is a CPU-accessible 1MB sliding window into VRAM, controlled by a single BAR0 register.
- Window placement is 64KB aligned; the 24-bit `BASE_ADDR` field addresses a 40-bit (1TB) VRAM
  space.
- nova-core's Rust implementation uses the type system to enforce exclusive access at compile time.
- The mutex protecting PRAMIN state must never be held across `GFP_KERNEL` allocations -- a
  subtle constraint documented directly in the code.
- Self-tests validate window auto-repositioning across 1MB boundaries before posting patches.
