📖 chapter one

What's a Ralph loop?

The simplest possible idea in agentic AI:

Keep handing the same prompt to the AI over and over until it decides it's finally done.

Iterative refinement, taken to its most ridiculous, beautiful extreme — and somehow it works.

📜 the canonical form

The dumbest possible thing

while :; do
  cat PROMPT.md | claude-code
done
Three lines. That's the whole technique. Everything else is hardening on top.
💡 why it works

Single-shot is hard. Loops are easy.

One prompt, one shot

Model has to nail it on the first try. P(success) per attempt = low.

One prompt, N tries

If P(success) is even 5%, after 100 loops you're at 99.4%.
1 − (1−0.05)100.

Persistence beats perfection.

🪝 hooks 101

What are Claude Code hooks?

User-defined shell commands that Claude Code runs automatically at lifecycle events.

SessionStart

fires when a session boots

UserPromptSubmit

before each user message

PreToolUse / PostToolUse

around each tool call

Notification

when Claude needs a permission

Stop ⭐

before the assistant turn ends

SubagentStop

before a subagent ends

Configured in ~/.claude/settings.json. The hook command gets a JSON payload on stdin and writes a JSON decision on stdout.
🔄 lifecycle

Lifecycle of one Claude turn

The Stop hook is the last gate before Claude actually stops. That's our hook.

⭐ the stop hook

What the Stop hook can do

It runs before Claude exits

Last chance to inspect what just happened.

It can override the stop

Reply with {"decision":"block","reason":"..."} and Claude keeps going with that reason as a new system message.

It receives a payload

JSON on stdin: session_id, transcript_path, cwd, model, ...

It must be fast

Synchronous. Anything slow stalls the turn ending.

🔬 anatomy

Stop hook in / out

📥 stdin (from Claude Code)

{
  "session_id": "abc123...",
  "transcript_path": "/.../session.jsonl",
  "cwd": "/home/joel/repo",
  "stop_hook_active": false
}

📤 stdout (your reply)

{
  "decision": "block",
  "reason": "task not yet verified —
             keep working"
}

Or empty/{} to allow the stop.

Giralph's giralph stop-hook subcommand is exactly this: read JSON in, decide, write JSON out.
⚠️ flaw 1

The judge is also the defendant

Existing Ralph loops let Claude decide whether Claude is done.

Self-assessment is a well-documented failure mode:

  • Claude says "tests pass" without running tests
  • Claude says "fixed!" with broken code on disk
  • Claude says "complete" because it wants to stop
⚠ self-graded
⚠️ flaw 2

State on disk = AI can wreck it

If your loop's "running" state is in .ralph/state.json, then:

# inside the loop, Claude has tool access:
$ rm -rf .ralph/
# loop controller wakes up, sees no state, exits clean
# AI escapes the budget without doing the work

The loop trusted the filesystem. The AI has the filesystem.

💥 unsafe
⚠️ flaw 3

Stop hooks in Bash are untestable

  • The official plugin's stop-hook is a multi-hundred-line Bash script
  • Has had real bash 3.x compatibility bugs
  • Edge cases in transcript parsing fail silently
  • You cannot unit-test JSON parsing in Bash without painful mocking
"I'll just sed-extract this JSON field" → 6 hours later
🐚 fragile
⚠️ flaw 4

No real verification pipeline

Existing tools leave verification entirely inside the prompt.

"Only emit the completion token when you have genuinely run the tests..."

There's no programmatic "actually run go test ./..." step. There's no independent reviewer. The prompt is the only line of defense.

🤷 no oracle
🦒 enter giralph

Giralph: the G is for greatly improved

🧠 Independent verifier

A separate Claude reads the transcript and grades the work.

🔒 Daemon-held state

State lives outside the AI's reach. Filesystem hacks don't work.

🧪 Compiled & tested

Single Go binary. Real unit tests. Not yet-another-bash-script.

A Ralph loop with brains, armor, and unit tests.

...is that a giraffe?

🔍 the killer feature

The verifier is a separate Claude

The promise is the AI's claim.
The transcript is the evidence.

🪙 Claim: "I fixed the flaky test."

📜 Evidence: No go test in the transcript.

⚖️ Verdict: Loop continues.

🔒 untouchable state

State lives in a daemon, not in a file

Why a daemon?

  • The AI has full FS access — it can rm -rf any file
  • A separate process is a different trust boundary
  • The AI must explicitly connect to its socket — no accidents

What it holds

  • Active sessions (keyed by Claude's session_id)
  • Retry budgets, iteration counts
  • Verification config per loop
  • Recent stop-hook timestamps (thrash detection)
🎬 example · step 1/8

1You start a loop

Inside an active Claude Code session, you type:

claude> /giralph-start Fix the flaky integration test --retry 50
→ registering loop with daemon...
→ loop registered: sid=abc123, retry=50, verify=agent

Slash command runs giralph start which IPC-calls the daemon.

🎬 example · step 2/8

2Claude works on the task

<Read internal/integration/flaky_test.go>
<Grep "WaitFor">
<Edit: replace timeout 100ms → 500ms>
<Bash: go test ./internal/integration/ -count=10>
PASS · 10/10 runs
<Assistant: "I've made the test more robust. Done.">

Normal Claude turn — file edits, tool calls, reasoning. Nothing giralph-specific yet.

🎬 example · step 3/8

3Claude emits a promise

Two paths exist. Either is fine. The one Claude is told about is the fence:

```ralph-promise
{"summary": "Increased flaky test timeout from 100ms to 500ms;
             test now passes 10/10.",
 "artifacts": ["/abs/path/internal/integration/flaky_test.go"]}
```

Or via Bash IPC: !giralph promise '{"summary": "...", ...}'

The promise is a claim. The daemon doesn't trust it yet.
🎬 example · step 4/8

4Claude is about to stop → Stop hook fires

Claude Code is finishing the turn. Before it exits, it pipes JSON to giralph stop-hook:

{
  "session_id": "abc123",
  "transcript_path": "/home/joel/.claude/projects/-repo-x/abc123.jsonl",
  "cwd": "/home/joel/repo/x",
  "stop_hook_active": false
}

The stop-hook binary forwards this over the Unix socket to the daemon.

🎬 example · step 5/8

5Daemon dispatches verification

  • Daemon looks up abc123 in its sessions map → finds the loop
  • Slices this loop's events out of the transcript JSONL → writes to a temp file
  • Spawns the verifier (acpx → Claude Code), passes $GIRALPH_TRANSCRIPT in env
  • Verifier reads the promise + transcript and decides PASS or FAIL
verifier prompt:
  Promise summary: "Increased flaky test timeout from 100ms..."
  Transcript: $GIRALPH_TRANSCRIPT
  → Did the AI actually do what it claimed?
🎬 example · step 6/8

6Verifier rejects (this time)

Suppose the AI didn't actually run go test:

verifier verdict: FAIL
reason: promise claims test passes 10/10, but no
        `go test` tool call appears in transcript.
daemon → stop-hook → Claude Code:
{"decision":"block", "reason":"verification failed: ..."}

Claude Code now doesn't stop. It continues the turn with the reason as a system message.

🎬 example · step 7/8

7Loop continues — Claude tries again

With the verifier's feedback in context, Claude actually runs the tests this time:

<Bash: go test ./internal/integration/ -count=10>
PASS · 10/10 runs
<Emit ralph-promise with new artifact list>

Stop hook fires again. Daemon dispatches verifier again.

Iteration counter ticks. If retry=50, the loop has 49 attempts left.
🎬 example · step 8/8

8Verified → loop ends

verifier verdict: PASS
reason: transcript shows `go test ... -count=10` succeeded
        immediately before the promise.
daemon: marking loop completed
daemon → stop-hook → Claude Code: {} (allow stop)
Claude exits cleanly.

🎉 Task done — and we have an independent reviewer's word on it.

🛡️ spec-005

Retry budget — exhaustion protection

Every loop is bounded by --retry N:

  • Default: 100
  • --retry 50: cap at 50
  • --retry 0: unlimited (opt-in)

When budget hits zero:

Loop.Iteration >= MaxRetries
→ status: "exhausted"
→ stop-hook returns: allow stop
→ daemon evicts the loop
A broken or stubborn AI cannot loop forever. The budget is the universal kill switch.
🛡️ spec-007

Password gate — modify & cancel

giralph modify and giralph cancel require the operator's local-account password, read from /dev/tty.

👤 Operator (terminal)

Has a controlling tty. Sees the password prompt. Types it. ✅

🤖 Claude (no tty)

No controlling terminal. Opening /dev/tty fails. ❌

Validated against the system password DB via PAM's setuid unix_chkpwd. The prompt itself is the gate.

🛡️ spec-009

The daemon — out-of-process state

  • Long-lived giralph daemon service (systemd / launchd)
  • Listens on a per-user 0600 Unix socket
  • All loop state held in-memory, keyed by session_id
  • Shared token + UID check on every IPC request
  • Unknown-session stop-hook payloads pass through cleanly (no false positives for unrelated Claude sessions)
No filesystem state file means no rm -rf bypass.
🛡️ spec-011

Loop-hack detection (thrash reset)

The attack

AI emits empty turns to drain the budget:

turn 1 (2s) STOP
turn 2 (3s) STOP
turn 3 (1s) STOP  ⚠️

The defense

3 stops in <60s →

iteration: 47 → 0
budget restored
loop survives 🦒

Stealth: the AI-facing reason text never mentions "thrash." The mechanism stays invisible.

🛡️ spec-012

Nag escalation — talk back smarter

Same nag 50 times = wasted tokens. The wording escalates by repeat count.

Tier 1 · ≤3

Polite reminder of the fence path.

Tier 2 · 4–9

+ a "did you know?" block with the IPC alternative.

Tier 3 · ≥10

+ an operator hint: "the loop has rejected your last 10 turns…"

A real run logged 797 identical nags before this spec. Never again.

🛡️ spec-013

Re-inject after auto-compaction

When Claude Code's context overflows, it compacts — replacing the working window with a Claude-written summary.

The summary preserves task progress, not the verbatim Step 0 contract. Claude forgets how to signal completion.

Daemon detects the compaction marker → next stop-hook prepends the contract.

⌨️ cli · install

giralph install <claude|codex>

One-time, idempotent setup that wires giralph into your AI CLI.

  • Installs the systemd unit (Linux) / launchd plist (macOS)
  • Merges a Stop-hook entry into ~/.claude/settings.json
  • Writes /giralph-start and /giralph-cancel slash commands
  • Creates ~/.giralph.json
Each subtask is independently idempotent — re-running prints "skipped: already up to date" for anything already done.
⌨️ cli · start

giralph start

Register a loop with the daemon. Normally invoked via the /giralph-start slash command.

# In a Claude session:
/giralph-start Fix the flaky test --retry 50

# Or directly (debugging):
giralph start --task "..." --retry 50 --verify-cmd "make check"

Auto-detects the session id from ~/.claude/projects/<cwd-hash>/.

⌨️ cli · promise

giralph promise <json>

The alternative way to signal completion (besides the markdown fence).

!giralph promise '{"summary":"fixed bug",
                   "artifacts":["/abs/file.go"]}'
  • Auto-detects session id
  • Sends summary + artifacts to the daemon over IPC
  • Useful when the markdown fence is awkward (e.g., final turn is short)
⌨️ cli · stop-hook

giralph stop-hook · internal

Reads stop-hook JSON from stdin, forwards to daemon, writes decision JSON to stdout.

You don't run this — Claude Code does, automatically, via the hook config that install wrote.

🛟 Fail-open: if the daemon is down, this exits cleanly with an empty decision so a broken giralph never bricks Claude Code.

⌨️ cli · modify

giralph modify <sid> 🔒

Tweak an in-flight loop without cancelling it.

giralph modify abc123 --retry 100
giralph modify abc123 --task "new task text"
giralph modify abc123 --verify-cmd "make check"

🔒 password-gated. The AI cannot bump its own retry budget.

⌨️ cli · cancel

giralph cancel <sid> 🔒

Stop a loop mid-flight.

$ giralph cancel abc123
[sudo-style] Password: ●●●●●●●●
loop abc123 cancelled (status: cancelled)

Removes the loop from the daemon's sessions map. Next stop-hook for that session passes through cleanly.

🔒 password-gated — same gate as modify.

⌨️ cli · uninstall

giralph uninstall <claude|codex>

Reverses install in the opposite order, idempotently.

  1. Disables and removes the service unit
  2. Deletes /giralph-start and /giralph-cancel commands
  3. Removes only giralph's hook entry from settings (other hooks left intact)

~/.giralph.json and runtime state are kept by default. Pass --purge to delete them too.

✅ when to use · 1/2

🚀 Use it for prototypes

Greenfield exploration. The thing has no users yet. You want a working blob, not a polished product.

  • Scratch a script that compiles
  • Try a library you've never used
  • Sketch a CLI to see if the UX makes sense
  • Validate a hypothesis before investing real engineering
✓ great fit
✅ when to use · 2/2

🐢 Use it with weaker / cheaper models

The model needs many tries to reach the goal — and each try is cheap. Quantity beats quality, statistically.

Haiku × 50 retries > one Opus shot, when you can verify the result.

∞ × 🐢 = 🏁

Slow + persistent > fast + flaky.

✓ great fit
🚫 when not to use · 1/2

🏗️ Don't use it to ship a product

Giralph chases the goal. It will gladly produce something that compiles, runs, and is ugly inside.

  • Code style, abstractions, design choices
  • Module boundaries, naming, organization
  • The "feel" of the codebase

These are hard to verify, so the loop will skip them. The loop optimizes for what the verifier can check.

✗ wrong tool
🚫 when not to use · 2/2

🎨 Don't use it for anything taste-based

UIs. Marketing copy. Visual design. Anything where the success metric is human taste.

The verifier can confirm the button exists.
It cannot confirm the button is beautiful.

If you can't write a check that fails when the answer is wrong, a loop won't save you — it'll just tell you confidently it's right.
✗ unverifiable
💸 how not to use

Don't pair it with expensive models

A loop is a token-burning machine. Every iteration re-reads context, re-tries, and the verifier costs more tokens on top.

Token spend over 100 iterations:

Haiku in a loop  cheap

Sonnet in a loop  moderate

Opus in a loop  😱 your wallet

Match the model to the verifiability. Cheap + persistent + verified = giralph's sweet spot.
